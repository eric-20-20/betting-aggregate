# Website Data Flow

How today's picks and tier assignments flow from the pipeline to the website.

## Source of Truth

The pipeline's source of truth for daily picks is:

```
data/plays/plays_YYYY-MM-DD.json     (NBA)
data/plays/ncaab/plays_YYYY-MM-DD.json  (NCAAB)
```

These files are generated by `scripts/score_signals.py` and contain:
- `meta` — date, signal counts, tier counts
- `plays[]` — all A/B/C-tier picks with full scoring data
- `filtered[]` — D-tier picks that failed quality gate (limited data: signal + wilson_score + gate_reason)

## Export Path

`scripts/export_web_data.py` reads the plays file and produces two outputs:

### Public (unauthenticated visitors)
```
web/public/data/picks/public_YYYY-MM-DD.json
```
Contains:
- `teaser_picks` — top 3 A-tier picks with full details (free preview)
- `locked_picks` — all other A/B/C picks as blurred summaries (matchup, tier, confidence only)
- `meta` — date, tier counts

### Private (subscribers)
```
web/data/private/picks_YYYY-MM-DD.json
```
Contains:
- `plays[]` — all A/B/C picks with full details (sanitized: source names replaced with consensus strength labels, ROI stripped, expert names removed)

### Reports (public)
```
web/public/data/reports/
  consensus_strength.json
  market_type.json
  by_stat_type.json
  top_trends_summary.json
  recent_trends.json
  tier_performance.json
```
Generated from `data/reports/` with sanitization (ROI stripped, source names removed).

### History (subscribers)
```
web/data/private/history/
  index.json
  history_YYYY-MM-DD.json
```
Graded historical picks with WIN/LOSS/PUSH outcomes.

## What the Website Reads

The Next.js frontend (`web/lib/data.ts`) reads from the filesystem:

| Function | File | Access |
|----------|------|--------|
| `getPublicPicks(date)` | `web/public/data/picks/public_{date}.json` | Public |
| `getFullPicks(date)` | `web/data/private/picks_{date}.json` | Subscribers |
| `getTierPerformance()` | `web/public/data/reports/tier_performance.json` | Public |
| `getHistoryDay(date)` | `web/data/private/history/history_{date}.json` | Subscribers |

## Tier Assignment

Tiers are assigned by `score_signals.py:assign_tier()`:

| Tier | Criteria |
|------|----------|
| A | Pattern match + Wilson >= 0.46, OR 4+ sources + Wilson >= 0.62, OR 5+ sources + Wilson >= 0.58 |
| B | Wilson >= 0.46 + (pattern OR 2+ sources) |
| C | Wilson >= 0.45 (marginal) |
| D | Below thresholds or failed quality gate (excluded from export) |

## Pipeline Sequence (run_daily.sh)

```
score_signals.py
    writes → data/plays/plays_YYYY-MM-DD.json

export_web_data.py
    reads  → data/plays/plays_YYYY-MM-DD.json
    writes → web/public/data/picks/public_YYYY-MM-DD.json
    writes → web/data/private/picks_YYYY-MM-DD.json

git push → Vercel redeploy

push_to_supabase.py → syncs signals/grades/plays to Supabase
```

## Where Stale/Cached Mismatch Can Happen

1. **Pipeline re-run without re-export**: If `score_signals.py` runs again (e.g., after late-arriving source data), the plays file updates but the web export files are stale until `export_web_data.py` runs.

2. **Export without deploy**: Web files are written to disk but not live until `git push` triggers Vercel rebuild.

3. **Supabase lag**: `push_to_supabase.py` runs after web export. Supabase tables may briefly differ from web JSON files.

4. **No editorial gate**: The export reads raw pipeline output directly. There is no mechanism to review, override, or approve picks before they go live.

## Sanitization Rules

The export strips all source-identifying information:
- Source names (action, betql, covers, etc.) → replaced with consensus strength labels (1_source, 2_source, 3+_source)
- Expert names → removed
- ROI fields → stripped
- Source-specific trends → filtered from reports

The website never reveals which specific sources agree on a pick.
